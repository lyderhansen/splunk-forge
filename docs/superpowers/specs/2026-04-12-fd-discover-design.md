# Design: fd-discover — Log format discovery skill

**Date:** 2026-04-12
**Status:** Draft — awaiting user review before implementation plan
**Author:** Brainstormed with Claude Code (superpowers:brainstorming)
**Based on:** `2026-04-11-discover-logformat-design.md` (validated TA-FAKE-TSHRT spec, 487 lines). This spec documents the deltas for the FAKE_DATA plugin port, not a full redesign.

---

## Context

The discover-logformat skill was designed, implemented, and validated in TA-FAKE-TSHRT during the 2026-04-11 planning session. Two plan iterations were executed (v1 offline MVP, v2 research-enabled), and canary tests passed against fortigate (description-only research), aws_cloudtrail (sample-based), and custom_internal_app (offline KV detection).

This spec ports that skill to the FAKE_DATA plugin as `fd-discover`. The core architecture (6 phases: input validation, research, analysis, confidence gates, artifact writing, handoff) is unchanged. This document specifies only what changes in the plugin context.

**Key design principle:** Research is the default mode, not sample analysis. Most users arrive with "I want Palo Alto logs" and nothing else. The skill must produce a usable SPEC from just a source name.

## Relationship to the original spec

The original spec (`2026-04-11-discover-logformat-design.md`) remains the source of truth for:
- Research loop algorithm (iterative, weakest-area-first)
- Format detection patterns (8-pattern ordered table)
- Field extraction rules per format (JSON flatten, KV split, CEF parse, etc.)
- Confidence scoring model (per-field frequency + vendor-doc bonus)
- SPEC schema shape (source, format, fields, sourcetype, generator_hints, research_metadata)
- Batch mode behavior
- Collision handling (overwrite / rerun / abort)

This spec overrides or extends the original on the 8 points listed below.

## Changes from TA-FAKE-TSHRT spec

### 1. Skill name: fd-discover

Invocation: `/fd-discover <source_id> [--sample=<path>] [--doc=<url>] [--description=<text>] [--no-search]`

Same flags as original. `--ta=<id>` for Splunkbase app ID is kept. `--interactive`, `--batch`, `--threshold`, `--min-sources`, `--max-research-time` are kept with same defaults.

### 2. SPEC format: Python module (SPEC.py)

Instead of YAML, the SPEC is written as a Python module with a single `SPEC` dict. This is consistent with the plugin's stdlib-only, Python-modules-for-config philosophy (no PyYAML dependency).

```python
"""Discovery spec for fortigate. Generated by fd-discover."""

SPEC = {
    "schema_version": 1,
    "generated_at": "2026-04-12T14:30:00Z",
    "generated_by": "fd-discover v0.1.0",

    "source": {
        "id": "fortigate",
        "display_name": "Fortinet FortiGate",
        "vendor": "Fortinet",
        "product": "FortiGate",
        "description": "Traffic logs from Fortinet FortiGate NGFW...",
    },

    "category": "network",
    "source_groups": ["network"],

    "format": {
        "type": "kv",
        "line_break": "\n",
        "timestamp": {
            "field": "date",
            "format": "%Y-%m-%d",
            "auxiliary_field": "time",
            "auxiliary_format": "%H:%M:%S",
        },
        "encoding": "utf-8",
        "confidence": 0.92,
    },

    "sourcetype": {
        "name": "fortinet:fortigate:traffic",
        "category_path": "network/fortigate_traffic.log",
        "confidence": 0.88,
    },

    "fields": [
        {"name": "srcip", "type": "ipv4", "required": True,
         "example": "10.10.30.55", "confidence": 0.95},
        # ...
    ],

    "sample_events": [
        {
            "raw": "date=2026-01-05 time=14:30:45 srcip=10.10.30.55 ...",
            "parsed": {"srcip": "10.10.30.55", "dstip": "185.220.101.42"},
            "source_url": None,
        },
    ],

    "generator_hints": {
        "suggested_module_name": "generate_fortigate",
        "suggested_function_name": "generate_fortigate_logs",
        "volume_category": "firewall",
        "baseline_events_per_day": 1000,
        "dependencies": [],
        "multi_file": False,
        "scenarios": {
            "existing": [],
            "proposed": [],
        },
    },

    "research_metadata": {
        "sources_consulted": [
            {"url": "https://docs.fortinet.com/...", "kind": "vendor_doc",
             "trust": 1.0, "retrieved_at": "2026-04-12T14:28:00Z"},
        ],
        "total_research_time_sec": 45,
        "overall_confidence": 0.87,
        "unresolved_questions": [],
    },
}
```

`fd-add-generator` reads this with: `from fake_data.discover.fortigate.SPEC import SPEC`.

### 3. Artifact placement: fake_data/discover/

All artifacts live under the workspace:

```
fake_data/
├── discover/
│   └── <source_id>/
│       ├── __init__.py      # empty, makes it importable
│       ├── SPEC.py           # machine-readable (for fd-add-generator)
│       ├── REPORT.md         # human-readable narrative + citations
│       ├── samples/          # log samples used in analysis
│       │   ├── vendor_doc_01.log
│       │   └── user_provided.log
│       └── research.json     # full audit trail (URLs, timestamps, tokens)
├── generators/
├── world.py
└── ...
```

Note: `__init__.py` in the discover/<source_id>/ directory is needed so Python can import `SPEC.py` as a module.

### 4. Research: subagent-based from the start

Instead of running research inline (polluting the main skill's context with large web-fetch results), fd-discover dispatches a research subagent from the start.

**Research subagent contract:**

Input (passed via Agent tool prompt):
- `source_id`
- `explicit_inputs` (sample path, doc URLs, description, TA ID)
- `budget` (max time in seconds, default 300)
- `no_search` flag

Output (returned in final message as structured text):
- `ResearchFindings` struct with: `samples_found[]`, `vendor_hint`, `product_hint`, `description_hint`, `field_hints[]`, `sources_consulted[]`, `elapsed_sec`

**Research priority order** (same as original, but with preset check first):

0. **Preset check** (before subagent dispatch) — check `presets/<source_id>.py` in plugin repo. If exists, load preset and skip research entirely.
1. **Explicit inputs** — `--sample`, `--doc`, `--ta`. Highest trust.
2. **Vendor docs search** — `WebSearch "<vendor> <source> log format site:docs.<vendor>.com"`. Most reliable external source.
3. **Splunkbase TA** — search for existing Splunk TA. Extract README, props.conf, field definitions. Extremely valuable when available.
4. **Sample log search** — `WebSearch "<source> sample logs"` to find example events published in documentation or training materials.
5. **Community forums** — Splunk Community, vendor forums. Only if steps 1-4 give < 0.6 confidence.
6. **Fallback generic search** — unrestricted WebSearch, lower trust.

**Subagent model selection:** Use sonnet for the research subagent. Research requires judgment about which search results are relevant, which pages to fetch, and how to extract structured data from unstructured web content.

### 5. Preset check: new phase before research

Before dispatching the research subagent, fd-discover checks `presets/<source_id>.py` in the plugin repo (relative path from SKILL.md: `../../../presets/<source_id>.py`).

If a preset exists:
- Read it (it contains a `PRESET` dict with the same shape as SPEC)
- Skip research entirely
- Proceed to Phase C (analysis) with preset data as the "findings"
- Set `research_metadata.sources_consulted = [{"kind": "preset", "trust": 1.0}]`
- Note in REPORT.md: "This discovery used a bundled preset. No web research was performed."

If no preset exists: proceed with research subagent as normal.

Presets directory is empty in X1. This mechanism is ready for when presets are added later.

### 6. Infrastructure suggestion in handoff

After writing artifacts, fd-discover checks the source's category against existing INFRASTRUCTURE in world.py:

1. Read `fake_data/world.py`, extract `INFRASTRUCTURE` list.
2. Determine what infrastructure role this source implies (e.g. category=network, source involves firewall → role=firewall).
3. Check if each location has that role in INFRASTRUCTURE.
4. If any location is missing the implied role, include a simple yes/no prompt in the handoff:

```
Discovery complete for 'palo_alto' (confidence: 0.89)

Artifacts: fake_data/discover/palo_alto/
  SPEC.py, REPORT.md, samples/, research.json

Next step:
  /fd-add-generator palo_alto

Your workspace has firewall devices at HQ1 but not OFF1.
Add a firewall to OFF1? [yes/skip]
```

If yes: add the infra entry to world.py directly (read, parse the INFRASTRUCTURE list, append, write back). If skip: note it in REPORT.md.

Category-to-infrastructure-role mapping:

| Source category | Implied infra role |
|---|---|
| network | firewall |
| cloud | proxy (if hybrid setup implied) |
| windows | directory_server |
| web | web_server |
| ot | plc |
| other | no suggestion |

### 7. Handoff points to /fd-add-generator

Phase F handoff message:

```
Discovery complete for '<source_id>' (confidence: <score>)

Artifacts written to fake_data/discover/<source_id>/:
  - SPEC.py        (machine-readable, for fd-add-generator)
  - REPORT.md      (human-readable summary)
  - samples/       (<N> log samples)
  - research.json  (audit trail)

Next step: review REPORT.md, then run:
  /fd-add-generator <source_id>
```

### 8. fd-add-generator integration

After fd-discover ships, `fd-add-generator` gains a third input mode:

| Mode | Trigger | What happens |
|---|---|---|
| Wizard | no --sample, no SPEC.py exists | Ask questions (existing behavior) |
| Sample | --sample=<path> | Run fd-discover internally with --no-search, read SPEC.py, scaffold |
| From SPEC | SPEC.py already exists in fake_data/discover/<source_id>/ | Read SPEC.py directly, scaffold |

When a user runs `/fd-add-generator fortigate` and `fake_data/discover/fortigate/SPEC.py` exists, fd-add-generator reads it automatically and skips the wizard. No `--from-spec` flag needed — just convention: if SPEC.py exists, use it.

This change to fd-add-generator is **noted here but implemented separately** — it is a small update to `fd-add-generator/SKILL.md` that adds "check for existing SPEC.py" at the start of Phase A.

## Phases summary (fd-discover)

| Phase | What | Changed from original? |
|---|---|---|
| A — Input validation | Parse flags, normalize source_id, check workspace | Skill name, workspace check via manifest.py |
| A.5 — Preset check | Check presets/ in plugin repo | **New** |
| B — Research | Dispatch subagent for web research | **Subagent-based** (was inline) |
| C — Analysis | Format detection, field extraction, sourcetype suggestion | Unchanged |
| D — Confidence gates | Auto-triggered questions for low-confidence areas | Unchanged |
| E — Artifact writing | Write SPEC.py, REPORT.md, samples/, research.json | SPEC.py format, fake_data/discover/ path |
| F — Handoff | Print summary, suggest infra, point to fd-add-generator | **Infra suggestion** new, skill names updated |

## Metadata-only fallback

When research finds vendor docs with field descriptions but no raw log examples (common for proprietary sources), the skill falls through to metadata-only mode:

- `format.type = "unknown"`, `format.confidence = 0.0`
- Fields populated from research field_hints at confidence 0.7
- Skip format detection entirely
- REPORT.md notes: "No sample events found. Fields derived from vendor documentation only."
- Overall confidence excludes format.confidence from the mean so 0.0 doesn't tank the score

This was validated in TA-FAKE-TSHRT's fortigate canary test and must be preserved in the port.

## Canary test

Three canary sources, same as TA-FAKE-TSHRT but adapted for plugin paths:

1. **Known source with sample** — `/fd-discover test_kv --sample=<fixture>` with a 3-line KV file. Assert: SPEC.py exists, format.type = "kv", fields extracted, confidence > 0.
2. **Known source, research only** — `/fd-discover fortigate --description="FortiGate NGFW traffic logs"`. Assert: SPEC.py exists, vendor = "Fortinet" or similar, at least 3 fields, research.json has > 0 sources.
3. **Unknown source, offline** — `/fd-discover custom_app --sample=<fixture> --no-search`. Assert: SPEC.py exists, format detected from sample, no research performed.

## Out of scope

- CIM alignment post-pass (future)
- props.conf/transforms.conf generation (fd-build-app)
- Updating fd-add-generator to read SPEC.py (separate small plan, noted above)
- Populating presets/ (separate effort)
- GitHub sample scraping (ruled out, same as original)

## Dependencies

- Requires a working FAKE_DATA workspace (run `/fd-init` first)
- WebSearch and WebFetch tools must be available for research mode
- Agent tool must be available for research subagent dispatch
